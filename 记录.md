# 查询系统优化记录

> 最后更新：2026-05-19

## 优化目标

1. **解决主线查询卡顿** — 数据量大时查询缓慢，怀疑一次性映射大量数据
2. **解决表格列宽拥挤** — 字段都挤在一起，需要适当的间距
3. **保留列字段分组翻页功能** — 点击上一页/下一页切换字段组，无水平滚动

---

## 一、查询性能优化

### 修改文件：`finddata.cpp`

#### `slot_Load_Data_show` — 改用 SQL_CALC_FOUND_ROWS

**问题：** 每次查询先执行 `SELECT COUNT(*)` 获取总行数，再执行 `SELECT * ... LIMIT/OFFSET` 获取数据。对于百万级数据的主线表，两次全表扫描是卡顿的根本原因。

**优化后：**

```cpp
// 之前：两次查询
SELECT COUNT(*) FROM datas_linecode_a1 WHERE ...       // 全表扫描一次
SELECT * FROM datas_linecode_a1 WHERE ... LIMIT 25 OFFSET 0  // 全表扫描第二次

// 之后：一次查询 + FOUND_ROWS()
SELECT SQL_CALC_FOUND_ROWS * FROM datas_linecode_a1 WHERE ... LIMIT 25 OFFSET 0
SELECT FOUND_ROWS()  // 仅读取一个数字，无表扫描
```

`SQL_CALC_FOUND_ROWS` 在同一个查询中同时计算总行数和返回数据，`FOUND_ROWS()` 仅读取 MySQL 内部计数器，**减少一次全表扫描**。

---

### 修改文件：`mainwindow.ui`

#### 默认每页行数从 50 → 25

减少单次加载的数据量，降低每次查询的网络传输和内存占用。用户仍可通过 `allPage` 控件自行调整（范围 10-10000）。

## 二、列宽自适应方案

### 修改文件：`mainwindow.h` / `mainwindow.cpp`

### 核心思路

不再使用 `QHeaderView::Stretch`（等宽分布，导致窄列和宽列一样宽），改为**基于内容宽度按比例分配**。

### 新增：列宽缓存机制

```cpp
// mainwindow.h 新增成员
QMap<QString, QVector<int>> m_cachedColumnWidths;  // 表名 -> 每列内容宽度缓存
```

### 新增函数：`cacheColumnWidths(QTableView *view)`

数据加载到 model 后自动执行：
1. 临时显示所有列，设为 `ResizeToContents` 模式
2. 测量每列的实际内容宽度
3. 缓存到 `m_cachedColumnWidths`
4. 恢复列隐藏状态和 `Interactive` 模式

### 重写：`columnPageSizeForView`

**之前：** 每次调用都切换 `ResizeToContents` + `processEvents`，强制完整布局重算，非常昂贵。

**之后：** 直接从缓存读取宽度，计算多少列能放入视口，无需布局重算。

### 重写：`applyColumnPageToView`

**之前：** `QHeaderView::Stretch` 等宽分布，导致字段挤在一起。

**之后：**
- 使用 `QHeaderView::Interactive` 模式
- 根据缓存的内容宽度按比例分配视口空间
- **宽列多分空间，窄列少分空间**，填满视口
- 列最小宽度 60px，防止被挤压到不可见
- 保持 `ScrollBarAlwaysOff`，字段分组翻页导航

```
示例：视口宽度 1200px，当前页有 3 列，内容宽度分别为 [80, 200, 120]
按比例分配：1200 * 80/400 = 240px, 1200 * 200/400 = 600px, 1200 * 120/400 = 360px
结果：窄列 240px，宽列 600px，中列 360px —— 合理分配，填满视口
```

## 三、列分组翻页修复

### 修改文件：`mainwindow.cpp`

### 问题：行分页与列分页冲突

**之前的问题：**
- `CurPage` 同时被行分页（`updatePageControls`）和列分页（`applyColumnPageToView`）修改，互相覆盖
- `Btn_PageUp`/`Btn_PageDown` 点击后触发 `CurPage::valueChanged` 信号，导致重新查库
- 列翻页后 CurPage 会变化，触发不必要的数据库查询

### 修复：

| 控件 | 修改前 | 修改后 |
|------|--------|--------|
| `CurPage` | 同时控制行偏移和列分组页码 | **仅显示列分组页码**（只读） |
| `CurPage_2` | 同上 | **仅显示列分组总页数**（只读） |
| `Btn_PageUp` | 触发 `CurPage` 修改 → 触发查库 | **直接调用 `applyColumnPageToView` 切换列分组**，不查库 |
| `Btn_PageDown` | 同上 | **同上** |

### 具体改动：

**`on_Btn_PageDown_clicked` / `on_Btn_PageUp_clicked`：**
```cpp
// 之前
ui->CurPage->setValue(curPage + 1);  // 触发 valueChanged → 重新查库

// 之后
applyColumnPageToView(view, info.columnPage + 1, true);  // 仅切换列显示
```

**移除 `CurPage::valueChanged` → `loadDataForCurrentPage` 信号连接：**
CurPage 不再触发数据库查询。

**`updatePageControls` 简化：**
不再覆盖 CurPage / CurPage_2 / Btn_PageUp / Btn_PageDown 的状态，仅保留 `applyColumnPageToCurrentView()`。

**`loadDataForCurrentPage` 始终使用 offset=0：**
行翻页功能已从界面移除（CurPage 改为列分页显示），每次查询始终从第 1 行开始。

### 2. 解决翻页异常 + 字段内容完整显示

**问题：** 重叠翻页逻辑（overlap=1）导致列边界错乱，`qMax` 宽度分配方式在 `allocated` 累加时溢出，使剩余列被挤到负宽度，出现"字段只显示一半"、"空列占一整行"。

**修复：**

**2a. 去掉列重叠，恢复清晰分页边界：**

```diff
- const int stride = qMax(1, pageSize - 1);
- const int startColumn = (currentPage - 1) * stride;
- const int totalPages = 1 + qMax(0, (columnCount - pageSize + stride - 1) / stride);
+ const int totalPages = qMax(1, (columnCount + pageSize - 1) / pageSize);
+ const int startColumn = (currentPage - 1) * pageSize;
```

**2b. 重写宽度分配，去掉 `qMax` 累加 bug：**

```cpp
// 修正后：内容窄于视口 → 按比例拉伸填满
double ratio = (double)viewportWidth / totalContentWidth;
int allocated = 0;
for (int vi = 0; vi < visibleIndices.size(); ++vi) {
    int col = visibleIndices[vi];
    int w;
    if (vi == visibleIndices.size() - 1) {
        w = viewportWidth - allocated;  // 最后一列拿剩余空间
    } else {
        w = (int)(cachedWidths[col] * ratio);
        allocated += w;
    }
    view->horizontalHeader()->resizeSection(col, w);
}

// 内容≥视口 → 直接使用内容宽度，不压缩，允许留白
for (int vi = 0; vi < visibleIndices.size(); ++vi) {
    view->horizontalHeader()->resizeSection(col, cachedWidths[col]);
}
```

比例分配使用**纯净的浮点乘法 + 最后一列兜底**，不再使用 `qMax` 绕过分配累加，确保各列宽度总和恒等于 `viewportWidth`。

## 第三次优化：标签页懒加载优化

### 修改文件：`mainwindow.cpp`

### 问题

- `dataLoaded` 在整个会话中只增不减，切换标签页后即使数据已过时也不会重新查询
- 列宽缓存 `m_cachedColumnWidths` 在重新查询时未清除，新旧数据列内容不同时宽度不适配

### 修改点

**1. `loadDataForCurrentPage` — 每次查库前清空标记和缓存：**

```cpp
PageInfo &pageInfo = m_pageInfoMap[m_currentTabKey];
pageInfo.dataLoaded = false;            // 新增：标记数据未加载
m_cachedColumnWidths.remove(m_currentTabKey);  // 新增：清除旧列宽缓存
pageInfo.columnsPerPage = ui->allPage->value();
```

确保任何触发 `loadDataForCurrentPage` 的操作（查询、切换标签页、allPage 改变）都会：
- 置 `dataLoaded = false`，让切换标签页时不跳过查库
- 清除列宽缓存，让数据到达后 `cacheColumnWidths` 重新测量

**2. `on_Btn_Find_clicked` — 清除列宽缓存：**

```cpp
m_cachedColumnWidths.remove(m_currentTabKey);  // 新增
resetCurrentPage();
loadDataForCurrentPage();
```

点击"查询"时清除旧列宽缓存，确保新查询结果使用正确宽度。

### 懒加载工作流

```
用户点击"查询" → dataLoaded=false → 查库 → 数据到达
  → modelReset → cacheColumnWidths（重新测量） → applyColumnPageToView
  → signal_totalPages → updatePageControls
  → dataLoaded=true

用户切换到其他标签页 → dataLoaded=true → 跳过查库
  → applyColumnPageToCurrentView（恢复列显示）
  
用户再切回来 → 本次会话中 dataLoaded 仍为 true → 跳过查库（缓存有效）
```
